Тз: Личный Кабинет Клиента Для Прозрачной Разработки
Техническое задание (ТЗ)
1. Общая информация

Название проекта: Client Project Cabinet (рабочее название)

Назначение системы: Веб-система для прозрачного ведения проектов разработки между разработчиком и клиентами.

Система решает проблемы:

недоверие клиента

непонимание статуса проекта

хаос в договорённостях

потеря ТЗ и правок

Ключевая идея: клиент видит «кухню проекта», но в контролируемом и понятном виде.

2. Пользовательские роли и доступы
2.1 Client (Клиент)

Может:

зарегистрироваться и войти в систему

создавать неограниченное количество проектов

просматривать статус каждого проекта

видеть этапы, ТЗ, сроки, файлы и комментарии

оставлять комментарии и вопросы

утверждать этапы и версии ТЗ

запрашивать связь с администратором

иметь несколько проектов одновременно

Не может:

видеть проекты других клиентов

изменять системные настройки

2.2 Admin (Разработчик / Менеджер)

Может:

видеть все проекты всех клиентов

управлять статусами проектов

управлять этапами (создание, редактирование, скрытие)

управлять сроками

редактировать и версионировать ТЗ

фиксировать утверждения клиента

управлять пользователями

получать уведомления о действиях клиентов

3. Авторизация и регистрация
3.1 Способ входа (MVP)

Email + Magic Link

Без паролей

3.2 Логика входа

Пользователь вводит email

Система отправляет Magic Link

По переходу по ссылке:

если пользователь новый → создаётся профиль Client

если пользователь существует → выполняется вход

4. Основные сущности и логика данных
4.1 User

id

name

email

phone (optional)

telegram (optional)

role (client / admin)

company_name (optional)

created_at

Один User может иметь несколько проектов.

4.2 Project

id

title

description

goal (бизнес-цель проекта)

type (site / crm / automation / ai / other)

status:

draft (создан, но не отправлен)

pending (ожидает связи)

active (в работе)

paused (ожидание клиента)

completed (завершён)

owner_id (User)

budget_min

budget_max

deadline (optional)

created_at

Логика:

проект создаётся клиентом

изначально имеет статус pending

после связи админ переводит в active

4.3 ProjectStage (Этап проекта)

id

project_id

title

description

order (порядок отображения)

status:

not_started

in_progress

review (на проверке клиентом)

approved (утверждён)

visible_to_client (boolean)

planned_start_date

planned_end_date

actual_end_date (optional)

Логика:

этапы формируют таймлайн проекта

клиент видит только visible_to_client = true

клиент может утвердить этап только в статусе review

4.4 TechnicalSpecification (ТЗ)

id

project_id

version (1.0, 1.1, 2.0)

content (markdown)

status:

draft

on_review

approved

created_at

approved_at (optional)

Логика:

ТЗ всегда версионируется

только одна версия может быть on_review

после утверждения версия становится approved

4.5 Comment

id

project_id

stage_id (optional)

author_id

content

type:

comment

question

fix

approval

created_at

Комментарий всегда привязан либо к проекту, либо к конкретному этапу.

4.6 File

id

project_id

stage_id (optional)

uploader_id

file_url

file_name

file_type

created_at

4.7 ActivityLog

id

project_id

user_id

action (string)

created_at

Используется для истории и защиты от споров.

5. Клиентская часть (Client UI)
5.1 Клиентский дашборд

Отображается после входа.

Содержит:

список проектов

фильтры по статусу

индикатор "Требуется ваш ответ"

кнопку "Создать проект"

Карточка проекта:

название

статус проекта

текущий активный этап

дата последнего обновления

5.2 Создание проекта (Client)

Форма:

Название проекта (обязательно)

Тип проекта

Краткое описание

Цель проекта

Желаемые сроки (optional)

Бюджет (диапазон)

Загрузка файлов

После отправки:

проект создаётся со статусом pending

администратор получает уведомление

5.3 Страница проекта (Client View)

Блоки:

Общая информация

Таймлайн этапов

ТЗ (последняя версия)

Комментарии

Файлы

История активности

Действия клиента:

оставить комментарий

задать вопрос

утвердить этап

утвердить ТЗ

запросить связь

6. Административная часть (Admin UI)
6.1 Админ-дашборд

Показывает:

новые проекты (pending)

проекты без ответа > 48 часов

этапы на проверке

проекты с риском по срокам

6.2 Управление проектом (Admin)

Администратор может:

изменить статус проекта

назначить этапы из шаблона

редактировать описание проекта

устанавливать дедлайны

ставить проект на паузу

6.3 Управление этапами

Для каждого этапа:

название

описание

порядок

статус

даты

видимость для клиента

Действия:

отправить этап на проверку

завершить этап

6.4 Управление ТЗ

Функционал:

создание новой версии ТЗ

редактирование markdown

отправка клиенту на проверку

фиксация утверждения

7. Уведомления
7.1 Email-уведомления

Отправляются при:

создании проекта

новом комментарии

этапе на проверке

запросе связи

8. UX-принципы

минимум кликов

одинаковая логика для всех проектов

не перегружать клиента деталями

важные действия всегда подтверждаются

9. Нефункциональные требования

Frontend: Next.js

Backend: NestJS

DB: PostgreSQL

Auth: Magic Link

UI: shadcn/ui

Hosting: Vercel + Supabase / Railway

10. MVP-ограничения

Не реализуется:

чаты в реальном времени

тайм-трекинг по часам

финансовые отчёты

сторонние интеграции

11. Критерии готовности MVP

Клиент может создать несколько проектов

Админ управляет этапами

Фиксация утверждений работает

История изменений сохраняется

Система снижает количество ручных коммуникаций
